Fixing the Dev Cycle

Problems

  • Miscommunication between devs about how to apply class names
  • Bad default styles being overwritten by a reset stylesheet
  • Selectors are buried too deep within markup
  • Complex and multiple selectors make rapid development a pain
  • Starting from scratch lengthens the time of the project
  • Trouble upgrading Compass and configuration

Solutions

Maintain Current Configuration

Which includes using Compass, Blueprint and our own reset stylesheet. This fixes little problems, but allows us to continue to use the most up-to-date Compass version released with each project.

Problems

  • Compass configuration headaches
  • Tough for developers to rapidly prototype
  • Need to jump into CSS to lay something out
  • Has a potential to bloat projects

Benefits

  • Configurable mixins and a basic familiarity
  • Actively developed, wrote about and expanded upon
  • Tons of plugins, such as Blueprint, Susy Grids, etc.
  • Documented

Use old-style Blueprint with custom Mixins

Leverage the old non-semantic Blueprint grid with non-semantic mixins which allow developers to include class names in their markup to reference layout.

Problems

  • Impossible to make responsive
  • Non-semantic markup
  • Blueprint is no longer under active development

Benefits

  • Familiarity for designers and developers
  • Good reset stylesheet

Twitter Bootstrap, straight up

Use the default Twitter Bootstrap configuration as a starting point for each new project.

Problems

  • Currently no SASS version
  • Need to overwrite a ton of styles for customization
  • Fear of stuff looking the same from site to site
  • Bloat of unused styles
  • Still required to use nested class names and memorization of classes

Benefits

  • Easy to get off the ground
  • Maintained by a ton of developers and designers
  • Documented
  • Standardized class names
  • Nice base CSS stylesheet

Developing our own

Develop an Ample Getting Started guide that normalizes all HTML elements and also includes our own custom mixins we deem necessary.

Problems

  • Need to constantly browser test new additions and mixins
  • A lot of work to get off the ground
  • Need to document

Benefits

  • Customizable to our needs
  • Debatable what to include and what to leave out
  • Lean CSS

Rolling our Own

Steal what we like from other frameworks Twitter bootstrap, Susy, Blueprint, etc and bake them into our own.

Problems

  • Takes awhile to setup
  • No documentation
  • When a new update comes out of each Framework, we need to manually update

Benefits

  • Ability to use tested CSS in addition to customizing it to suit our needs
  • Writing our own mixins allows us to only use what we need and leave out the rest = lean CSS

Onto the Grid

Susy vs. Blueprint

"Susy is also different because it is not possible without Compass. Blueprint is a port, Susy is native. Because of that, it seems to me that Blueprint doesn't take advantage of the full potential of Sass - but still leans on outdated notions of what a framework is. Susy uses the full power of Sass to allow absolutely any type of grid (fixed,fluid,elastic) and even combinations, specializing in the useful and difficult fluid-elastic, without any change at all in how you use it. Susy is simply a set of tools that do math for you, and that's how it should be used."

Going forward, I think the more we continue (if we continue) to do responsive designs, Susy is a better option if we stick with Compass.

Next Steps?

No matter what we decide to use, we should continue to maintain an Ample-Getting-Started repo with a base set of styles and setup to easily allow copying and pasting of the JavaScript, Stylesheet and basic application.html.erb layout file for each new project we start. This repo will hold the latest version of the framework and also serve as a place we can add our own mixins.
